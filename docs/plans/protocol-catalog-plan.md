# Protocol Catalog — Implementation Plan

## What

`scry catalog` command that auto-generates a protocol reference from saved Arena games. Produces per-type YAML files + an index README + a shared dictionary of magic numbers. Lives in `docs/protocol/` in the repo.

## Decisions made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Where does the command live? | scry-ts (`scry catalog`) | Same corpus, same analysis engine, same card resolver |
| Regeneration model | Full regen every time | Corpus is small (<100 games), variance parses everything in <1s |
| What triggers regen? | Manual (`scry catalog`) | Human reviews diff, commits what makes sense |
| Human notes | In-file `notes:` field preserved on regen | One file per type, data + context together |
| What gets committed | YAML + auto-rendered index README | Per-type YAML is readable. One index for GitHub browsing. No per-type markdown. |
| Conformance cross-ref | Deferred | Ship data layer first, add Kotlin grep coverage later |
| Detail key dictionary | Separate `dictionaries.yaml` | counter_type, category, damage_type cross-cut annotation types |

## Per-type YAML shape

```yaml
# auto-generated by scry catalog — do not edit except `notes:` field
type: CounterAdded
instances: 23
games: 8
persistence: transient
always_keys: [counter_type, transaction_amount]
sometimes_keys:
  - key: some_rare_key
    pct: 15
value_samples:
  counter_type: [1, 139, 42]
  transaction_amount: [1, 2, 3]
cards:
  count: 9
  notable:
    - { grp: 84262, name: "Enduring Bondwarden", context: "counter_type=1 (backup)" }
    - { grp: 86770, name: "Unexpected Fangs", context: "counter_type=139 (lifelink)" }
co_types: [PowerToughnessModCreated, Counter]
first_seen: 2026-03-29
last_seen: 2026-04-04
notes: |
  Fires when a counter is placed on a permanent. counter_type values
  map to specific counter kinds — see dictionaries.yaml.
```

The `notes:` field is the only hand-editable part. Generator reads existing file, preserves `notes:`, overwrites everything else.

## Card list strategy

Auto-collect which cards triggered each annotation via affectorId → resolve to grpId.

Surface differently by category:
- **Structural** (Designation, PhaseOrStepModified): `cards: { count: N }` — card list not interesting
- **Mechanic-linked** (CounterAdded, AbilityWordActive): `cards: { notable: [...] }` — card → detail mapping
- **Interaction** (DamageDealt, ZoneTransfer): category breakdown, not card list

Heuristic: if a detail key has >3 distinct values across games → interesting axis → include notable cards. Always-same value → structural → skip card list.

## Dictionaries

Flat YAML keyed by field name → value → human label. Auto-grows as new values observed. Seeded from what we know today:

- `counter_type`: +1/+1, -1/-1, loyalty, poison, stun, lifelink, etc.
- `zone_transfer_category`: CastSpell, Countered, Exile, SBA_Damage, etc. (21 known from rosetta.md Table 2)
- `damage_type`: 1=combat, 2=noncombat
- `designation_type`: 1=Commander
- `promptId`: 2=priority, 6=attackers, 7=blockers, 10=targets, 34=mulligan, 86=ordering, 144=commander-zone
- `mana_color`: int→color mapping
- `action_type`: ActionType enum → description

New values auto-added with `null` label on regen. Human fills in the label.

## Implementation approach

1. **Data collection** — extend variance engine to also collect prompt types and action types (currently only annotations). Add card resolution per annotation instance.
2. **YAML writer** — per-type YAML with round-trip `notes:` preservation. Read existing file → keep notes → overwrite data.
3. **Dictionary builder** — scan all detail values, seed known mappings, add unknowns as `null`.
4. **Index renderer** — summary tables in markdown, dictionary-enriched where applicable.
5. **CLI** — `scry catalog` with flags: `--dry-run` (preview), `--diff` (show changes since last run), `--markdown` (just re-render index without re-analyzing).

## Testing

- Golden test: run catalog on reference-brawl.log, verify one annotation type YAML matches expected shape.
- Round-trip test: write YAML with notes, regen, verify notes preserved.
- Dictionary test: known counter_type values resolve correctly.

## What this does NOT do (deferred)

- Conformance coverage (`--coverage` grep against Kotlin source)
- Incremental regeneration
- Cross-file hyperlinks in markdown
- Replacing existing prompt-specs/*.md (they stay as interpretation layer)
