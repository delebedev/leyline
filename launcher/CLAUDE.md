# launcher

Tauri v2 desktop app — player-facing wrapper around the leyline server. One-click start, Arena config management, restore to stock.

## Stack

- **Backend:** Rust (Tauri v2, std::process for sidecar, reqwest for health check)
- **Frontend:** vanilla TypeScript + Vite + Bun (no framework)
- **Sidecar:** jlink-bundled leyline server from `just bundle` (embedded as Tauri resource)

## Architecture

**Not a formal Tauri sidecar.** The jlink bundle is a directory tree (jre/ + lib/ + bin/leyline), not a single binary. Tauri's `externalBin` expects target-triple-named binaries. Instead: bundle as Tauri resources via staging dir, Rust spawns `bin/leyline` via `std::process::Command`.

```
launcher/
├── src-tauri/
│   ├── src/main.rs          Tauri setup, command registration, graceful shutdown
│   ├── src/server.rs        Sidecar lifecycle — spawn, health poll, kill, DB management
│   ├── src/arena.rs         MTGA detection, config deploy/restore, launch
│   ├── tauri.conf.json      Window config, version (CalVer), icon
│   └── .bundle-stage/       Build-time staging dir (gitignored, created by just launcher-build)
│       └── leyline/         Full sidecar: bin/ lib/ jre/ res/ data/
├── src/
│   ├── index.html           App shell
│   ├── main.ts              All frontend logic — state machine, Tauri invoke calls
│   └── style.css            Dark theme, teal accent
└── resources/
    ├── services.conf         Localhost server config (deployed to Arena)
    └── NPE_VO.bnk           Stub audio bank (56 bytes)
```

## Resource Bundling (Critical Knowledge)

**Tauri v2 resolves ALL resource paths from `src-tauri/` (Cargo manifest dir), not CWD.**

The `just launcher-build` recipe:
1. Runs `just bundle` → `build/bundle/` (jre + lib + bin)
2. Copies bundle to `src-tauri/.bundle-stage/leyline/`
3. Extracts `res/` from archive (card data, locales) into staging dir
4. Copies `data/player.db` (seed database) into staging dir
5. Runs `bun tauri build --config '{"bundle":{"resources":[".bundle-stage/leyline/",".bundle-stage/changelog.md"]}}'`

**Key gotchas** (each cost CI iterations to discover):
- `TAURI_CONFIG` env var does NOT exist in Tauri v2. Use `--config` CLI flag.
- Glob `**/*` matches files. `**` alone matches only directories.
- Map format `{"glob":"target/"}` flattens directory structure. Array format `["path/"]` preserves it.
- Array format with `../../` produces `_up_/_up_/` in resources. Use a staging dir instead.
- `upload-artifact@v4` excludes dotfiles by default (`include-hidden-files: true` to fix).

## Sidecar Resolution

1. **Built app:** resource dir → `.bundle-stage/leyline/bin/leyline`
2. **Dev fallback:** walks up from CWD to find `build/bundle/` at repo root

If neither exists, Start shows an error. Run `just bundle` before `bun tauri dev` for full e2e.

## Sidecar CWD

The sidecar's working directory matters — it writes `logs/`, `data/`, `recordings/` relative to CWD.

- **Production:** `resource_dir/.bundle-stage/leyline/` (self-contained bundle with toml, data/, res/)
- **Dev:** repo root (has `leyline.toml`, `data/player.db`, `logs/`)
- **NEVER `src-tauri/`** — Tauri's file watcher monitors `src-tauri/` and restarts the app on any file change. Sidecar writing logs there causes an infinite restart loop.

## Player Database Lifecycle

- **Golden DB**: `launcher/resources/player.db` — committed fallback, used when `data/player.db` doesn't exist
- **Seed DB**: `data/player.db` — generated by `just seed-db`, preferred over golden if present
- **Runtime DB**: `~/Library/Application Support/dev.leyline.launcher/player.db`
- **First launch**: launcher copies seed from bundle to app data dir
- **Server**: receives path via `LEYLINE_PLAYER_DB` env var
- **No migration**: if schema changes, user must delete and re-seed (future improvement)
- **Known limitation**: golden DB can go stale if schema/decks change (tracked: leyline-971)

## Server Logs

Sidecar stdout/stderr → `~/Library/Logs/dev.leyline.launcher/leyline-server.log`. Always check this when Start fails.

## Testing

- `cargo test` — 6 TLS cert generation tests (CA, signing, chain, expiry)
- `just launcher-smoke` — headless integration test: generates temp certs, spawns sidecar with `--cert`/`--key`, verifies health + TLS cert chain on FD and AccountServer. No OS trust needed, no UI. Runs in release CI after `launcher-build`. Requires `just bundle` first.
- `just launcher-check` — static bundle check (files exist, DMG size)

## Debugging the launcher

- **Always use tmux** for `bun run tauri dev` — background tasks get killed, taking the launcher with them.
- **Check the sidecar's own log** at `~/Library/Logs/dev.leyline.launcher/leyline-server.log`, not just Tauri stderr. Tauri command errors go to the frontend, not stderr.
- `tmux new-session -d -s launcher "bun run tauri dev 2>&1 | tee /tmp/launcher.log"`

## Key modules

**server.rs** — `ServerState` enum (`Stopped | Starting | Running | Error`). `start_server` resolves bundle dir, generates TLS certs, copies seed DB, spawns sidecar with `--cert`/`--key` + `LEYLINE_PLAYER_DB`, polls `http://127.0.0.1:8091/health` every 500ms. State changes emitted as Tauri events.

**tls.rs** — Local CA + server cert generation via `rcgen`. `ensure_certs()` is called before sidecar spawn — generates CA, trusts it in OS keychain/store, signs server certs. Platform trust: macOS `security`, Windows `certutil`, Linux NSS `certutil` (`~/.pki/nssdb`). Tests: `cargo test` in `src-tauri/` — 6 tests covering CA generation, signing, chain files, expiry detection.

**arena.rs** — `detect_arena` scans known paths per platform (macOS Epic/Steam, Windows Epic/Steam, Linux Steam/Proton). `deploy_config` copies services.conf + NPE_VO.bnk, sets platform-specific prefs. `restore_arena` reverses everything. Linux uses same Windows file layout (Proton) and launches MTGA via `steam://rungameid/2141910`.

**main.ts** — No framework. DOM manipulation, `invoke()` for Tauri commands, `listen()` for server state events. Play button toggles Start/Stop.

## Build Commands

```bash
just launcher-build        # full build: gradle → bundle → stage → Tauri → check
just launcher-build-fast   # skip gradle, reuse existing bundle artifacts (after first build)
just launcher-check        # verify staging dir + DMG (runs automatically after build)
just launcher-dev          # hot-reload dev mode (run `just bundle` first for sidecar)
```

## Local Testing Loop

**Always test locally before tagging a release.**

```bash
# 1. Build everything (launcher-check runs automatically at the end)
just launcher-build
# Or, if you already built once and only changed Rust/TS/config:
just launcher-build-fast

# 2. Install from DMG
DMG=$(ls launcher/src-tauri/target/release/bundle/dmg/*.dmg)
hdiutil attach "$DMG"
cp -R /Volumes/Leyline/Leyline.app /Applications/
xattr -cr /Applications/Leyline.app
hdiutil detach /Volumes/Leyline

# 3. Kill any existing leyline processes
pkill -f leyline-launcher; pkill -f leyline.LeylineMainKt
for p in 30010 30003 8090 8091; do lsof -ti :$p | xargs kill -9 2>/dev/null; done

# 4. Launch app and click Start via osascript
open /Applications/Leyline.app
sleep 3
osascript -e 'tell application "System Events" to tell process "leyline-launcher" to click button "Start" of group 2 of UI element 1 of scroll area 1 of group 1 of group 1 of front window'

# 5. Wait for server and verify
sleep 5
curl -s http://127.0.0.1:8091/health   # {"status":"ok"}
cat ~/Library/Logs/dev.leyline.launcher/leyline-server.log  # check for exceptions
ls ~/Library/Application\ Support/dev.leyline.launcher/player.db  # DB exists
```

Common failures:
- `launcher-check` reports 6MB DMG → `--config` flag not working, or `just bundle` wasn't run
- "damaged" error on open → forgot `xattr -cr`
- Red flash then Stopped → check server.log, usually port conflict or missing `res/`
- "no such table: players" → seed DB not bundled, server using in-memory fallback

## Config management

Mirrors `just dev-setup` / `just dev-teardown` from root justfile:

| Action | What it does |
|--------|-------------|
| Deploy | Copy services.conf + NPE_VO.bnk to StreamingAssets, `defaults write CheckSC=0` |
| Restore | Remove services.conf, `defaults delete CheckSC` |

## Versioning

CalVer: `vYYYY.M.D`. Three files must match (checked by `just release-check`):
- `src-tauri/tauri.conf.json` → `.version`
- `package.json` → `.version`
- `index.html` reads version from Tauri API at runtime (no hardcoded string)

## Adding features

Frontend is intentionally minimal — vanilla TS, no component system. If the UI grows beyond ~3 screens, consider adding Svelte. For now, keep it flat.

Settings that affect the server (format, opponent, auto-pass) will need a config file or CLI flags on the sidecar. Not implemented yet — hardcoded defaults.

## Platform support

- **macOS** arm64 — .dmg, tested
- **Windows** x64 — NSIS installer, tested
- **Linux** x64 — AppImage, builds but needs real hardware testing (Steam Deck / Proton)

Linux: MTGA runs under Proton (Wine), so Arena paths use Windows layout. TLS cert trust uses NSS `certutil` (`~/.pki/nssdb`). MTGA launched via `steam://rungameid/2141910`. Sidecar CWD uses app data dir (AppImage filesystem is read-only).
