# launcher

Tauri v2 desktop app — player-facing wrapper around the leyline server. One-click start, Arena config management, restore to stock.

## Stack

- **Backend:** Rust (Tauri v2, std::process for sidecar, reqwest for health check)
- **Frontend:** vanilla TypeScript + Vite + Bun (no framework)
- **Sidecar:** jlink-bundled leyline server from `just bundle` (embedded as Tauri resource)

## Architecture

**Not a formal Tauri sidecar.** The jlink bundle is a directory tree (jre/ + lib/ + bin/leyline), not a single binary. Tauri's `externalBin` expects target-triple-named binaries. Instead: bundle as Tauri resources via staging dir, Rust spawns `bin/leyline` via `std::process::Command`.

```
launcher/
├── src-tauri/
│   ├── src/main.rs          Tauri setup, command registration, graceful shutdown
│   ├── src/server.rs        Sidecar lifecycle — spawn, health poll, kill, DB management
│   ├── src/arena.rs         MTGA detection, config deploy/restore, launch
│   ├── tauri.conf.json      Window config, version (CalVer), icon
│   └── .bundle-stage/       Build-time staging dir (gitignored, created by just launcher-build)
│       └── leyline/         Full sidecar: bin/ lib/ jre/ res/ data/
├── src/
│   ├── index.html           App shell
│   ├── main.ts              All frontend logic — state machine, Tauri invoke calls
│   └── style.css            Dark theme, teal accent
└── resources/
    ├── services.conf         Localhost server config (deployed to Arena)
    └── NPE_VO.bnk           Stub audio bank (56 bytes)
```

## Resource Bundling (Critical Knowledge)

**Tauri v2 resolves ALL resource paths from `src-tauri/` (Cargo manifest dir), not CWD.**

The `just launcher-build` recipe:
1. Runs `just bundle` → `build/bundle/` (jre + lib + bin)
2. Copies bundle to `src-tauri/.bundle-stage/leyline/`
3. Extracts `res/` from archive (card data, locales) into staging dir
4. Copies `data/player.db` (seed database) into staging dir
5. Runs `bun tauri build --config '{"bundle":{"resources":[".bundle-stage/leyline/",".bundle-stage/changelog.md"]}}'`

**Key gotchas** (each cost CI iterations to discover):
- `TAURI_CONFIG` env var does NOT exist in Tauri v2. Use `--config` CLI flag.
- Glob `**/*` matches files. `**` alone matches only directories.
- Map format `{"glob":"target/"}` flattens directory structure. Array format `["path/"]` preserves it.
- Array format with `../../` produces `_up_/_up_/` in resources. Use a staging dir instead.
- `upload-artifact@v4` excludes dotfiles by default (`include-hidden-files: true` to fix).

## Sidecar Resolution

1. **Built app:** resource dir → `.bundle-stage/leyline/bin/leyline`
2. **Dev fallback:** `../build/bundle/bin/leyline` (from `just bundle`)

If neither exists, Start shows an error. Run `just bundle` before `bun tauri dev` for full e2e.

## Player Database Lifecycle

- **Seed DB**: `data/player.db` — generated by `just seed-db`, bundled in `.bundle-stage/leyline/data/`
- **Runtime DB**: `~/Library/Application Support/dev.leyline.launcher/player.db`
- **First launch**: launcher copies seed from bundle to app data dir
- **Server**: receives path via `LEYLINE_PLAYER_DB` env var
- **No migration**: if schema changes, user must delete and re-seed (future improvement)

## Server Logs

Sidecar stdout/stderr → `~/Library/Logs/dev.leyline.launcher/leyline-server.log`. Always check this when Start fails.

## Key modules

**server.rs** — `ServerState` enum (`Stopped | Starting | Running | Error`). `start_server` resolves bundle dir, copies seed DB, spawns sidecar with `LEYLINE_PLAYER_DB`, polls `http://127.0.0.1:8091/health` every 500ms. State changes emitted as Tauri events.

**arena.rs** — `detect_arena` scans known macOS paths (Epic/Steam), returns path + storefront source. `deploy_config` copies services.conf + NPE_VO.bnk, sets `CheckSC=0`. `restore_arena` reverses everything.

**main.ts** — No framework. DOM manipulation, `invoke()` for Tauri commands, `listen()` for server state events. Play button runs: deploy_config → start_server → wait for healthy.

## Local Testing Loop

**Always test locally before tagging a release.**

```bash
# 1. Build everything
just launcher-build

# 2. Verify .dmg size (~90MB = good, ~6MB = resources missing)
ls -lh src-tauri/target/release/bundle/dmg/*.dmg

# 3. Mount and inspect
hdiutil attach src-tauri/target/release/bundle/dmg/*.dmg
ls /Volumes/Leyline/Leyline.app/Contents/Resources/.bundle-stage/leyline/
# Must contain: bin/ lib/ jre/ res/ data/

# 4. Install
cp -R /Volumes/Leyline/Leyline.app /Applications/
xattr -cr /Applications/Leyline.app
hdiutil detach /dev/diskN

# 5. Kill any existing leyline processes
pkill -f leyline-launcher; pkill -f leyline.LeylineMainKt
for p in 30010 30003 8090 8091; do lsof -ti :$p | xargs kill -9; done

# 6. Launch and test
open /Applications/Leyline.app
# Click Start, wait for Running state

# 7. Verify
curl -s http://127.0.0.1:8091/health   # {"status":"ok"}
cat ~/Library/Logs/dev.leyline.launcher/leyline-server.log  # check for exceptions
ls ~/Library/Application\ Support/dev.leyline.launcher/player.db  # DB exists
```

Common failures at each step:
- Step 2: 6MB → `--config` flag not working, or `just bundle` wasn't run
- Step 4: "damaged" error → forgot `xattr -cr`
- Step 6: red flash then Stopped → check server.log, usually port conflict or missing `res/`
- Step 7: "no such table: players" → seed DB not bundled, server using in-memory fallback

## Config management

Mirrors `just dev-setup` / `just dev-teardown` from root justfile:

| Action | What it does |
|--------|-------------|
| Deploy | Copy services.conf + NPE_VO.bnk to StreamingAssets, `defaults write CheckSC=0` |
| Restore | Remove services.conf, `defaults delete CheckSC` |

## Versioning

CalVer: `vYYYY.M.D`. Three files must match (checked by `just release-check`):
- `src-tauri/tauri.conf.json` → `.version`
- `package.json` → `.version`
- `index.html` reads version from Tauri API at runtime (no hardcoded string)

## Adding features

Frontend is intentionally minimal — vanilla TS, no component system. If the UI grows beyond ~3 screens, consider adding Svelte. For now, keep it flat.

Settings that affect the server (format, opponent, auto-pass) will need a config file or CLI flags on the sidecar. Not implemented yet — hardcoded defaults.

## Platform support

macOS arm64 only (dev preview). Cross-platform via CI matrix (Windows/Linux) is planned — jlink cross-builds with target-platform JDKs, Tauri builds on each runner.
